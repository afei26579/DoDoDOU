# 色号系统与库存开发计划

更新时间：2026-05-14

## 当前进度概览

当前项目已经从“色号系统基础”推进到“库存管理基础可用”的早期版本。

色号系统已经具备稳定品牌 key、运行时色号映射、品牌色卡查询、按品牌色号反查颜色等基础能力。图纸结果页已经开始使用 `PatternColorRequirement` 作为物料清单模型，为库存检查、替代色和后续扣库存流水预留了结构。

库存管理已经新增独立 IndexedDB store 和“我的库存”页面，结果页物料清单可以读取库存并计算够用、缺少或未知状态。库存逻辑保持在图纸生成算法之外，作为图纸用色统计的增强层。

### 4. 图纸结果页库存检查

- 图纸结果页物料清单会读取用户库存。
- 根据 `PatternColorRequirement` 和库存数据计算：
  - 需要数量
  - 已有数量
  - 缺少数量
  - `enough`
  - `missing`
  - `unknown`
- 没有录库存时，不影响图纸生成、查看和下载。
- 对 `MANMAN G11`、`MANMAN S4` 这类重复码，库存计算已改成同品牌同色号共享同一份剩余库存，避免重复高估。

关键文件：

- `src/features/beads/model/inventoryStore.ts`
- `src/pages/workshop/components/WorkshopResultStatsSheet.tsx`

### 5. 色号校验调整

- `npm run color-system:validate` 当前可以通过。

- 当前已知重复码：

  - `MANMAN S4`：同一个漫漫色号暂时对应不同颜色，当前按业务现状允许。

## 当前产品边界

- 色号系统是基础能力。
- 库存系统是增强能力。
- 没有录库存的用户，也必须能正常生成、编辑、查看和下载图纸。
- 库存逻辑不能塞进图纸生成算法。
- 替代色不能静默自动应用，必须用户确认。
- 扣库存不能自动发生，必须用户确认。
- 图纸应保留原始推荐色，同时允许后续记录用户实际替代后的用色。

## 推荐下一步

建议下一步优先继续 P3 的“缺色清单 / 补货清单”：

1. 在结果页物料清单中单独汇总 `missing` 项。
2. 生成按品牌、色号、缺少数量聚合的补货清单。
3. 给补货清单增加复制或导出能力。
4. 保持无库存用户仍然只看到普通物料清单。

原因：

- 当前库存 store 和结果页库存检查已经接通。
- 缺色清单是库存检查的自然延伸。
- 它不需要先实现替代色或扣库存，风险较低。

第二优先级可以做 P4 替代色推荐：

1. 先做同 hex 跨品牌推荐。
2. 再做用户库存中的近似色推荐。
3. 最后做用户手动替代和替代关系保存。

第三优先级再做 P5 扣库存和流水：

1. 先定义 `InventoryTransaction`。
2. 再做确认弹窗。
3. 然后支持扣减和撤销。

## 重要实现原则

- 图纸生成算法只负责生成图纸。
- 色号系统负责品牌色号、hex、色卡映射。
- 库存系统只围绕图纸用色统计做增强。
- 替代色必须用户确认。
- 扣库存必须用户确认。
- 重复码暂时允许，但库存计算必须避免重复高估。
