# 图片上传 → 转图纸 → 图纸展示核心算法说明

> 参考 `perler-beads` 项目的图片转拼豆流程，并结合当前 `Dodoudou` 的工坊页/参数页结构整理。

## 1. 目标

这条链路的目标不是简单“把图片缩小并换色”，而是把任意图片转换成一张**可制作、可预览、可编辑、可导出的拼豆图纸**。核心要兼顾三件事：

1. **视觉还原**：尽量保留原图主体、轮廓和色彩关系。
2. **制作可行**：减少噪点、碎色块、孤立点和过多颜色。
3. **参数可控**：允许用户通过尺寸、品牌、风格、容色阈值影响结果。

---

## 2. 对应到当前项目的流程

结合当前页面结构，流程可以理解为：

`上传图片` → `裁剪/预处理` → `参数设置` → `图纸生成` → `图纸预览/编辑` → `拼豆模式/下载`

### 页面职责映射

- `发现页 / 工坊页`：进入图片转图纸流程。
- `裁剪页`：完成图片输入与基础裁切。
- `工坊参数设置页`：配置画布尺寸、品牌色板、风格、颜色合并阈值。
- `图纸预览页`：展示算法输出结果，支持下载与编辑。
- `编辑页`：手动修正局部颜色或结构。
- `拼豆模式页`：基于最终图纸进行沉浸式制作。

当前 `WorkshopPage` 与 `WorkshopSettingsPage` 中已经有这些参数的 UI 雏形：尺寸、品牌、风格、容色/颜色合并等，它们就是算法输入的控制面。

## 4. 核心数据结构

建议把图纸结果拆成“像素级数据”和“统计级数据”两层。

```ts
interface PatternCell {
  x: number;
  y: number;
  colorId: string;      // 算法内部统一色号
  vendorCode: string;   // 具体品牌色号，如 MARD / COCO / 漫漫 / 盼盼 / 咪小窝
  hex: string;
}

interface PatternResult {
  width: number;
  height: number;
  cells: PatternCell[];
  palette: Array<{
    colorId: string;
    hex: string;
    vendorCodes: Record<string, string>;
    count: number;
  }>;
  stats: {
    totalCells: number;
    colorCount: number;
    tinyRegionCount: number;
    edgeNoiseCount: number;
  };
}
```

### 为什么要分两层

- **算法层**只关心统一颜色体系，便于计算。
- **展示层**再映射到不同品牌的色号，便于前端展示和导出。

## 5. 图片上传阶段

### 5.1 文件读取

用户上传后先做基础校验：

- 文件类型：`jpg / png / webp`
- 文件大小：限制过大的图片，避免前端爆内存
- 图像方向：读取 EXIF，必要时旋转修正

### 5.2 上传状态保存

当前页面里已经有 `localStorage` 的占位逻辑（`dodoudou:uploaded-image`）。后续建议保存：

- 文件名
- 宽高
- 裁剪框
- 预览缩略图
- 最近一次的处理参数

这样可以支持“继续上次项目”。

---

## 6. 裁剪与尺寸标准化

### 6.1 裁剪目的

裁剪不是简单美观处理，而是为了让后续图纸更稳定：

- 避免无效背景占用格子
- 让主体尽量落在中心区域
- 控制最终图纸的宽高比

### 6.2 标准化尺寸

图纸生成前要把输入图按目标网格尺寸重采样。例如：

- 新手：`72` 或 `96` 宽
- 标准：`96` 或 `120` 宽
- 细节优先：`120` 以上

尺寸越大，细节越多，但颜色碎片和制作难度也越高。

---

## 7. 预处理：去噪、平滑、增强轮廓

这一步决定图纸是否“干净”。

### 7.1 去噪

优先使用轻量去噪：

- 中值滤波：抑制椒盐噪点
- 双边滤波：尽量保留边缘

原则是：**去掉无意义噪点，但不要抹平主体边界**。

### 7.2 颜色空间转换

颜色计算建议不要直接在 RGB 中做最近色匹配，而是转到更适合感知比较的空间：

- RGB → Lab
- 再用 CIEDE2000 做色差比较

这样更符合“人眼看起来像不像”，而不是“数值上像不像”。

---

## 8. 网格化采样

图纸本质上是一个离散网格。采样阶段要把连续图像转成格子。

### 8.1 基本思路

将图片按目标尺寸切成小块，每个块对应一个拼豆格子。

每个格子的取色策略可分三种：

- **均值色**：适合平滑区域
- **主导色**：适合纹理复杂区域
- **自适应混合**：默认推荐

### 8.2 自适应策略

当一个块内部颜色变化不大时，用均值色；当块边缘变化明显时，用主导色。

可理解为：

- 低纹理 → 平滑处理
- 高纹理 → 保轮廓处理

这样能减少“整张图变脏”的问题。

---

## 9. 调色板量化

这是整个流程的核心。

### 9.1 统一色库

算法层需要一个标准色库作为基准。仓库中的 `perler-beads/色号对应表.csv` 已经说明了一个重要事实：

- 同一个标准色，可能对应多个品牌的不同色号
- 因此需要**统一基准色 → 品牌色号**的两层映射

### 9.2 最近色匹配

对于采样得到的每个格子颜色：

1. 转换为 Lab
2. 计算与标准色库每个颜色的色差
3. 选择最近的标准色
4. 再映射到用户选定品牌的色号

### 9.3 建议的色差策略

- 低成本版本：RGB 欧氏距离
- 推荐版本：Lab + CIEDE2000

推荐版本更适合图纸，因为拼豆图纸最怕“看着差不多，做出来很怪”。

---

## 10. 品牌色号映射逻辑

`色号对应表.csv` 的结构是：

- 第一行：品牌名
- 后续每行：统一标准色号，对应各品牌色号

例如一个标准色 `A01`，在不同品牌下可能分别对应：

- `MARD`
- `COCO`
- `漫漫`
- `盼盼`
- `咪小窝`

### 使用方式

- 算法先确定标准色 `standardColorId`
- 再根据用户选择的品牌读出对应的 `vendorCode`

这样可以保证：

- 图纸算法不依赖某个品牌
- 同一张图纸可以按不同品牌导出

---

## 11. 制作可行性优化

这一步是“从好看变成能做”的关键。

### 11.1 孤立点清理

如果某个格子周围 8 邻域大部分都不是它的颜色，那么它很可能是噪点。

处理方式：

- 用邻域主色替换
- 或合并到色差最小的相邻区域

### 11.2 小连通域合并

对每种颜色做连通域分析：

- 面积过小的区域
- 形状过碎的区域
- 边缘孤岛

这些区域优先合并到最近的大区域。

### 11.3 边缘保护

不能为了去碎点把主体轮廓抹掉。对于高对比边界，要降低合并力度。

### 11.4 颜色数控制

当最终颜色数过多时，应该做全局合并：

- 保留视觉主色
- 合并相近的辅助色
- 优先减少不必要的颜色碎片

这一步与 UI 中的“颜色合并阈值”直接对应。

---

## 12. 风格参数如何影响算法

当前 UI 里的“写实 / 动漫 / 极简”可以直接映射到不同处理权重。

### 写实

- 更保留细节
- 允许更多颜色
- 合并力度较弱

### 动漫

- 强化轮廓
- 更强调主色块
- 合并中等

### 极简

- 强化平面化
- 更少颜色
- 更强的色块统一

## 17. 参考项目核心逻辑：逐步函数拆解（含输入/输出）

下面按照 `perler-beads` 项目真实代码链路，把“图片上传 → 图纸生成 → 图纸预览”拆成可追踪的函数步骤。

### 17.1 颜色系统初始化

#### 函数：`getAllHexValues()` / `getMardToHexMapping()` / `loadFullColorMapping()`

来源：`src/utils/colorSystemUtils.ts`

**作用**
- 从 `colorSystemMapping.json` 中读取完整色号映射表。
- 建立“标准色 / 品牌色号”的基础映射。

**输入**
- `colorSystemMapping.json` 内的完整颜色映射对象。

**输出**
- `getAllHexValues()`：所有标准 `hex` 列表。
- `getMardToHexMapping()`：`MARD` 色号到 `hex` 的映射。
- `loadFullColorMapping()`：`Map<hex, { MARD, COCO, 漫漫, 盼盼, 咪小窝 }>`。

**说明**
- 这一层是整个图纸算法的颜色字典来源。
- 后续所有像素比色，最终都要回到这套映射表。

---

### 17.2 色板转换到指定品牌

#### 函数：`convertPaletteToColorSystem(palette, colorSystem)`

来源：`src/utils/colorSystemUtils.ts`

**作用**
- 把算法层的标准调色板，转换成用户当前选择品牌的展示色号。

**输入**
- `palette`: `PaletteColor[]`
- `colorSystem`: `MARD | COCO | 漫漫 | 盼盼 | 咪小窝`

**输出**
- 转换后的 `PaletteColor[]`
- 每个颜色的 `key` 变成当前品牌对应色号

**说明**
- 这个函数不改变 `hex`，只改变展示用的 `key`。
- 适合在 UI 和导出层使用。

---

### 17.3 颜色获取与展示

#### 函数：`getDisplayColorKey(hexValue, colorSystem)` / `getColorKeyByHex(hexValue, colorSystem)` / `convertColorKeyToHex(displayKey, colorSystem)`

来源：`src/utils/colorSystemUtils.ts`

**作用**
- 在 `hex`、品牌色号、展示键之间互相转换。

**输入**
- `hexValue` 或 `displayKey`
- `colorSystem`

**输出**
- 对应品牌的色号，或对应的 `hex`
- 找不到时返回 `?`

**说明**
- 图纸渲染时，格子可能显示色号。
- 编辑时，用户可能点选品牌色号再反查 `hex`。

---

### 17.4 基础颜色计算

#### 函数：`hexToRgb(hex)`

来源：`src/utils/pixelation.ts`

**作用**
- 把 `#RRGGBB` 转成 RGB 数值对象。

**输入**
- `hex: string`

**输出**
- `RgbColor | null`

**说明**
- 调色板与像素色都要先转成 RGB，才能算距离。

#### 函数：`colorDistance(rgb1, rgb2)`

来源：`src/utils/pixelation.ts`

**作用**
- 计算两个 RGB 颜色之间的欧氏距离。

**输入**
- `rgb1: RgbColor`
- `rgb2: RgbColor`

**输出**
- `number`

**说明**
- 这是当前项目里最直接的最近色距离算法。
- 后续可升级到 Lab + DeltaE。

#### 函数：`findClosestPaletteColor(targetRgb, palette)`

来源：`src/utils/pixelation.ts`

**作用**
- 在调色板中找到与目标颜色最接近的颜色。

**输入**
- `targetRgb: RgbColor`
- `palette: PaletteColor[]`

**输出**
- `PaletteColor`

**处理逻辑**
1. 遍历调色板。
2. 对每个候选色调用 `colorDistance()`。
3. 取距离最小者。
4. 如果完全匹配则提前结束。

**说明**
- 这是“像素色 → 拼豆色”的关键函数。
- 如果调色板为空，会返回黑色兜底色。

---

### 17.5 单个网格块代表色计算

#### 函数：`calculateCellRepresentativeColor(imageData, startX, startY, width, height, mode)`

来源：`src/utils/pixelation.ts`

**作用**
- 计算一个网格块的代表色。
- 支持两种策略：平均色、主导色。

**输入**
- `imageData: ImageData`
- `startX, startY, width, height`
- `mode: PixelationMode`
  - `Average`
  - `Dominant`

**输出**
- `RgbColor | null`

**处理逻辑**
1. 遍历块内所有像素。
2. 跳过透明像素（`alpha < 128`）。
3. 若为 `Average`，累计 RGB 后求平均。
4. 若为 `Dominant`，统计颜色频次，取出现最多的颜色。
5. 若块内没有有效像素，返回 `null`。

**说明**
- `Average` 更平滑，适合真实风格。
- `Dominant` 更干净，适合卡通/块面风格。

---

### 17.6 整张图的网格化生成

#### 函数：`calculatePixelGrid(originalCtx, imgWidth, imgHeight, N, M, palette, mode, t1FallbackColor)`

来源：`src/utils/pixelation.ts`

**作用**
- 这是核心入口函数。
- 把整张原图转成 `N × M` 的拼豆网格。

**输入**
- `originalCtx`: 原图 Canvas 2D 上下文
- `imgWidth`, `imgHeight`: 原图尺寸
- `N`: 横向格子数
- `M`: 纵向格子数
- `palette`: 当前调色板
- `mode`: 像素化模式
- `t1FallbackColor`: 兜底颜色

**输出**
- `MappedPixel[][]`

**处理逻辑**
1. 通过 `getImageData()` 读取原图像素。
2. 按 `N × M` 把图像切块。
3. 每个块调用 `calculateCellRepresentativeColor()`。
4. 若块无有效像素，写入透明/外部格子。
5. 若有代表色，调用 `findClosestPaletteColor()` 找最近色。
6. 最终返回二维网格。

**输出结构含义**
- 每个格子包含：
  - `key`: 色号
  - `color`: hex

**说明**
- 这是“上传图像 → 拼豆图纸”的主干函数。
- 后续图纸预览，就是把这个二维数组画到画布上。

---

### 17.7 可制作性优化（参考测试流水线）

`perler-beads` 的测试管线里还包含一层“生成后优化”。这部分在 `src/lib/test_image_to_pattern_pipeline.ts` 中更完整。

#### 函数：`test_mergeSimilarColors({ grid, palette, threshold })`

**作用**
- 合并相近颜色，减少图纸颜色碎片。

**输入**
- `grid`: 已生成的图纸网格
- `palette`: 调色板
- `threshold`: 色差阈值

**输出**
- 合并后的 `TestMappedCell[][]`

**处理逻辑**
1. 统计每种颜色出现次数。
2. 按使用量排序。
3. 逐对比较颜色距离。
4. 若色差小于阈值，则把较少的颜色统一替换为主色。

**说明**
- 这一步对应“颜色合并阈值”的 UI 参数。
- 适合在生成后做全局统一。

---

### 17.8 预览渲染

#### 函数：`test_generatePreviewCanvas(grid, canvas, dimensions)`

**作用**
- 把图纸网格绘制成预览图。

**输入**
- `grid`: 图纸数据
- `canvas`: 目标画布
- `dimensions`: `{ columns, rows }`

**输出**
- 无返回值，直接绘制到 canvas

**处理逻辑**
1. 计算每个格子的像素尺寸。
2. 清空画布。
3. 遍历所有格子。
4. 按颜色填充格子并绘制网格边界。
5. 若格子是外部/透明区域，则使用专门的背景色显示。

**说明**
- 这是图纸展示层的核心。
- 它不负责算法，只负责把算法结果画出来。

### 17.9 从上传到预览的完整函数链

把参考项目的核心逻辑串起来，实际上是这条链：

1. **载入颜色系统**
   - `loadFullColorMapping()`
   - 输出：完整品牌映射表

2. **生成调色板**
   - 由品牌映射表构造 `palette`
   - 输出：可参与比色的标准调色板

3. **读取原图像素**
   - `originalCtx.getImageData()`
   - 输出：`ImageData`

4. **单元格代表色提取**
   - `calculateCellRepresentativeColor()`
   - 输出：每个网格块的 RGB 代表色

5. **最近色匹配**
   - `findClosestPaletteColor()`
   - 输出：最接近的拼豆色号

6. **网格组装**
   - `calculatePixelGrid()`
   - 输出：`MappedPixel[][]`

7. **相近颜色合并**
   - `test_mergeSimilarColors()`（参考测试实现）
   - 输出：更适合制作的网格

8. **预览绘制**
   - `test_generatePreviewCanvas()`
   - 输出：可视化图纸

### 17.10 每一步的输入/输出总表

| 步骤 | 函数 | 输入 | 输出 |
|---|---|---|---|
| 颜色系统读取 | `loadFullColorMapping()` | 映射 JSON | `Map<hex, brandCodes>` |
| 色板品牌转换 | `convertPaletteToColorSystem()` | `palette + colorSystem` | 品牌化调色板 |
| Hex 转 RGB | `hexToRgb()` | `#RRGGBB` | `RgbColor` |
| 距离计算 | `colorDistance()` | 两个 RGB | 距离数值 |
| 最近色查找 | `findClosestPaletteColor()` | 目标 RGB + palette | 最近颜色对象 |
| 网格代表色 | `calculateCellRepresentativeColor()` | 单块 ImageData | `RgbColor | null` |
| 全图像素网格 | `calculatePixelGrid()` | 原图 ctx + 尺寸 + palette | `MappedPixel[][]` |
| 颜色合并 | `test_mergeSimilarColors()` | 网格 + 阈值 | 合并后的网格 |
| 图纸预览 | `test_generatePreviewCanvas()` | 网格 + canvas | 画布渲染结果 |

## 18. 结论

参考项目的核心并不是单个“像素化”函数，而是一条完整链路：

- 颜色系统映射
- 网格采样
- 最近色匹配
- 可制作性优化
- Canvas 预览渲染

把这些函数按输入/输出记录下来之后，后续你就可以直接把它们迁移到当前项目的“上传图片 → 图纸生成 → 图纸展示”流程中。
