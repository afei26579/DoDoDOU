# 登录系统设计方案与开发计划

## 1. 背景与目标

当前项目是无登录状态的创作工具，用户可以在浏览器本地完成图纸生成、编辑、收藏、库存录入和拼豆进度管理。现有数据形态如下：

- 前端：`React + Vite + React Router`。
- 后端：`Express + Prisma + SQLite`，当前主要承担公共画廊读取和发布接口。
- 本地数据：我的作品、编辑草稿、库存主要存储在浏览器 `IndexedDB` / `localStorage`。
- 画廊作者：目前只有 `authorId` / `authorName` 字段，不是真实用户身份。

登录系统的目标不是单独增加一个登录页，而是为后续能力建立统一身份层：

- 用户账号与个人资料。
- 我的图纸、草稿、拼豆进度云同步。
- 库存管理按用户归属。
- 收藏、发布、审核、评论等社区行为按用户归属。
- 订阅服务、会员权益、用量统计。
- 管理后台和内容审核权限。

## 2. 产品原则

### 2.1 保留游客模式

登录不应该阻断当前核心创作流程。建议保留游客可用能力：

- 浏览发现页和公共画廊。
- 上传图片生成图纸。
- 本地保存作品和草稿。
- 本地库存试用。
- 下载图纸。

需要登录的能力：

- 云同步作品和库存。
- 多设备访问。
- 发布作品到社区画廊。
- 收藏同步。
- 订阅、会员权益、AI 额度。
- 账号资料和历史记录。

### 2.2 先身份，后权益

第一期只建立稳定账号体系，不把订阅判断直接写死在登录逻辑里。登录只回答“是谁”，订阅和权限由独立权益层回答“能做什么”。

### 2.3 本地数据不丢失

现有用户数据在本机 IndexedDB 中。登录上线后必须支持本地数据迁移：

- 未登录时继续写本地。
- 登录后识别本地库存、图纸、草稿。
- 用户确认后同步到账号。
- 同步成功后服务端成为主数据源，本地作为缓存。

## 3. 推荐认证方案

### 3.1 第一阶段推荐：Cookie Session

推荐使用 `HttpOnly + Secure + SameSite` Cookie 保存会话，不建议把 JWT 放到 `localStorage`。

原因：

- 当前项目已有 Express 后端，Session 方案实现直接。
- `HttpOnly` Cookie 能避免前端脚本直接读取 token。
- 后续库存、订阅、发布审核都需要服务端鉴权。
- Session 可主动失效，适合封禁、退出登录、设备管理。

会话形态：

- 登录成功后服务端创建 `Session`。
- 数据库存储 `tokenHash`，不存明文 token。
- 浏览器保存 `dodoudou_session` Cookie。
- 前端通过 `GET /api/auth/me` 获取当前用户。

### 3.2 登录方式分阶段

第一期：

- 邮箱 + 密码登录。
- 邮箱验证码登录可作为密码登录的替代方案。

第二期：

- 手机号验证码。
- 微信登录。

第三期：

- 账号绑定与解绑。
- 多登录方式合并账号。
- 设备管理。

## 5. 数据模型设计

### 5.1 用户与会话

```prisma
enum UserRole {
  user
  admin
}

enum UserStatus {
  active
  disabled
  deleted
}

model User {
  id           String     @id @default(cuid())
  email        String?    @unique
  username     String?    @unique
  phone        String?    @unique
  passwordHash String?
  name         String?
  avatarUrl    String?
  role         UserRole   @default(user)
  status       UserStatus @default(active)
  createdAt    DateTime   @default(now())
  updatedAt    DateTime   @updatedAt

  sessions      Session[]
  oauthAccounts OAuthAccount[]
  inventory     BeadInventoryItem[]
  projects      WorkshopProject[]
  subscriptions Subscription[]
}

model Session {
  id         String   @id @default(cuid())
  userId     String
  tokenHash  String   @unique
  userAgent  String?
  ipAddress  String?
  expiresAt  DateTime
  createdAt  DateTime @default(now())
  updatedAt  DateTime @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId])
  @@index([expiresAt])
}

model OAuthAccount {
  id                String   @id @default(cuid())
  userId            String
  provider          String
  providerAccountId String
  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([provider, providerAccountId])
  @@index([userId])
}
```

## 6. API 设计

### 6.1 认证接口

```text
GET    /api/auth/me
POST   /api/auth/register
POST   /api/auth/login
POST   /api/auth/logout
POST   /api/auth/password/request-reset
POST   /api/auth/password/reset
```

注册和登录请求使用 `account` 字段，可填写邮箱或用户名。用户名长度至少 8 位，只允许字母、数字、下划线，且不能为纯数字。注册密码需要二次确认，密码至少 6 位，且不能为纯数字。

如果第一期选择验证码登录，则增加：

```text
POST   /api/auth/otp/request
POST   /api/auth/otp/verify
```

响应示例：

```json
{
  "user": {
    "id": "user_xxx",
    "email": "name@example.com",
    "username": null,
    "name": "豆豆",
    "avatarUrl": null,
    "role": "user",
    "status": "active"
  }
}
```

### 6.2 库存接口

```text
GET    /api/inventory
POST   /api/inventory
PATCH  /api/inventory/:id
DELETE /api/inventory/:id
POST   /api/inventory/sync
```

`/sync` 用于登录后批量上传本地库存：

- 服务端按 `userId + brandKey + code` 合并。
- 默认以后提交的数据为准。
- 如果服务端已有较新 `updatedAt`，返回冲突列表。

### 6.3 项目接口

```text
GET    /api/projects
POST   /api/projects
GET    /api/projects/:id
PATCH  /api/projects/:id
DELETE /api/projects/:id
POST   /api/projects/sync
```

第一期同步策略：

- 本地 `projectId` 存入 `clientProjectId`。
- 服务端生成自己的 `id`。
- 冲突时保留两份，避免误覆盖图纸。

### 6.4 画廊接口改造

当前 `POST /api/gallery/publish` 需要改造：

- 必须登录。
- 不再接受前端传入 `authorId`。
- 服务端使用 `req.user.id` 作为作者归属。
- 普通用户发布默认进入 `pending_review`。
- 管理员或官方账号才可直接 `published`。

建议新增：

```text
GET    /api/gallery/my-items
POST   /api/gallery/items/:id/favorite
DELETE /api/gallery/items/:id/favorite
POST   /api/admin/gallery/items/:id/approve
POST   /api/admin/gallery/items/:id/reject
POST   /api/admin/gallery/items/:id/offline
```

### 7.2 AuthProvider

职责：

- 应用启动时调用 `GET /api/auth/me`。
- 保存 `user`、`status`、`isAuthenticated`。
- 提供 `login`、`logout`、`refreshMe`。
- 统一处理 401 状态。

状态建议：

```ts
type AuthStatus = 'loading' | 'anonymous' | 'authenticated';
```

### 7.3 路由守卫

新增 `ProtectedRoute`：

- 未登录访问强登录页面时跳转 `/login?redirect=...`。
- 对可游客试用的页面，不强制跳转，只展示登录提示。

建议需要强登录的页面：

- `/account`
- `/subscription`
- 未来的 `/settings`

建议弱登录提示的页面：

- `/workshop/inventory`
- `/collection?tab=my`
- 发布到画廊弹窗

### 7.4 UI 入口

建议在以下位置加入账号入口：

- 发现页顶部右侧：头像 / 登录按钮。
- 作品集“我的”页：未登录时展示同步提示。
- 库存页：未登录可录入本地库存，但顶部提示“登录后可云同步”。
- 发布作品弹窗：发布前要求登录。

## 8. 本地数据迁移方案

### 8.1 迁移触发时机

登录成功后执行：

1. 检查 IndexedDB 中是否有本地项目。
2. 检查 IndexedDB 中是否有本地库存。
3. 检查 localStorage 中是否有收藏。
4. 如果存在，展示同步确认。

### 8.2 迁移流程

```text
User logs in
  -> AuthProvider gets user
  -> LocalMigrationService scans local data
  -> show migration dialog
  -> upload inventory sync
  -> upload projects sync
  -> upload favorites sync
  -> mark local migration completed for this user
```

本地标记：

```text
dodoudou.migration.completed.<userId> = true
```

### 8.3 冲突策略

库存：

- 相同 `brandKey + code` 合并。
- 数量默认以后提交为准。
- 如果双方都有修改时间，较新的覆盖较旧的。

作品：

- 相同 `clientProjectId` 更新同一服务端项目。
- 如果服务端项目更新时间更晚，保留两份并提示用户。

收藏：

- 取并集。

## 10. 安全要求

第一期必须完成：

- 密码使用 `argon2id` 或 `bcrypt` 哈希。
- Session token 只存 hash。
- Cookie 使用 `HttpOnly`。
- 生产环境 Cookie 使用 `Secure`。
- Cookie 设置 `SameSite=Lax`。
- 登录、注册、验证码接口限流。
- 登录失败次数限制。
- CORS 白名单。
- 所有写接口使用 `requireAuth`。
- 不信任前端传入的 `userId`、`authorId`、`role`。

第二期增强：

- 邮箱验证。
- 密码重置邮件。
- CSRF token 或双重提交 Cookie。
- 设备管理和远程退出。
- 审计日志。

下轮目标：在当前统一用户 ID 之上建立会员权益和用量统计的基础层，但不做完整支付闭环。

## 13. 第一版不建议做的内容

为控制复杂度，第一版不建议同时做：

- 完整管理后台。
- 多第三方登录同时上线。
- 复杂组织/团队权限。
- 图纸实时协同编辑。
- 订阅支付闭环。
- 服务端细粒度拆分所有图纸字段。

第一版重点是账号、会话、库存同步和作品同步的稳定闭环。

## 14. 推荐优先级

最高优先级：

1. `User` / `Session`。
2. `/api/auth/me`、登录、退出。
3. 前端 `AuthProvider`。
4. 库存同步。
5. 作品同步。

中优先级：

1. 画廊发布绑定用户。
2. 收藏同步。
3. 个人主页。
4. 审核状态。

低优先级：

1. 第三方登录。
2. 设备管理。
3. 订阅支付。
4. 管理后台。

## 15. 关键验收标准

登录系统完成后，至少应满足：

- 用户可以注册、登录、退出。
- 刷新页面后仍能识别当前用户。
- 未登录用户仍能使用核心创作能力。
- 登录后本地库存和作品可以迁移到账号。
- 服务端写接口不再信任前端传入的用户身份。
- 发布作品、库存、项目、收藏都能归属到真实用户。
- 后续订阅服务可以基于同一用户 ID 扩展。
