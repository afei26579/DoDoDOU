## 拼豆豆图纸数据实时保存方案设计

这类**高频读写 + 需要持久化**的场景，核心矛盾是：用户每画一笔都触发保存，但频繁写 IndexedDB 会造成性能问题。

---

### 核心思路：内存优先 + 分层持久化

```
用户操作
   ↓
内存状态 (zustand/useReducer)   ← 即时响应，毫秒级
   ↓
防抖写入 IndexedDB              ← 500ms~1s 后批量写
   ↓
（可选）导出/云端备份
```

---

### 一、状态管理层

用 **Zustand** 管理画布状态，所有操作只改内存：

```typescript
// store/patternStore.ts
interface PatternState {
  pixels: Map<string, string>   // "x,y" -> color
  history: PixelSnapshot[]      // 撤销栈
  future: PixelSnapshot[]       // 重做栈
  isDirty: boolean              // 是否有未保存的变更
}
```

`pixels` 用 `Map<"x,y", color>` 而不是二维数组，**只存有颜色的格子**，稀疏存储更省内存。

---

### 二、撤销/重做设计

每次操作记录 **diff（增量）** 而非全量快照：

```typescript
type PixelDiff = {
  type: 'paint' | 'erase' | 'fill'
  changes: Array<{
    key: string       // "x,y"
    before: string | null
    after: string | null
  }>
}
```

这样撤销栈不会随画布变大而膨胀，50步历史也只有少量数据。

---

### 三、IndexedDB 写入策略

关键：**不要每次操作都写 DB，用防抖合并写入**

```typescript
// hooks/useAutoSave.ts
const SAVE_DELAY = 800 // ms

export function useAutoSave(patternId: string) {
  const pixels = usePatternStore(s => s.pixels)
  const isDirty = usePatternStore(s => s.isDirty)
  const timerRef = useRef<ReturnType<typeof setTimeout>>()

  useEffect(() => {
    if (!isDirty) return

    // 清除上一个定时器，合并多次操作为一次写入
    clearTimeout(timerRef.current)
    timerRef.current = setTimeout(async () => {
      await db.patterns.put({
        id: patternId,
        pixels: Object.fromEntries(pixels), // Map → 普通对象存储
        updatedAt: Date.now()
      })
      usePatternStore.setState({ isDirty: false })
    }, SAVE_DELAY)

    return () => clearTimeout(timerRef.current)
  }, [pixels, isDirty])
}
```

### 四、IndexedDB Schema 设计

```typescript
// db/schema.ts  (用 Dexie.js 简化操作)
class BeadDB extends Dexie {
  patterns!: Table<PatternRecord>
  
  constructor() {
    super('BeadPatternDB')
    this.version(1).stores({
      patterns: 'id, updatedAt'
      // 不要把 pixels 加索引，数据量大
    })
  }
}

interface PatternRecord {
  id: string
  name: string
  width: number
  height: number
  pixels: Record<string, string>  // "x,y": color
  updatedAt: number
  thumbnail?: string              // base64 缩略图，用于列表预览
}
```

### 五、页面关闭时的兜底保存

防抖有延迟，用户可能在 800ms 内关闭页面：

```typescript
useEffect(() => {
  const handleBeforeUnload = () => {
    if (!isDirty) return
    // beforeUnload 里只能用同步 API
    // 用 localStorage 做临时缓冲
    localStorage.setItem(
      `unsaved_${patternId}`,
      JSON.stringify(Object.fromEntries(pixels))
    )
  }
  window.addEventListener('beforeunload', handleBeforeUnload)
  return () => window.removeEventListener('beforeunload', handleBeforeUnload)
}, [pixels, isDirty])

// 页面加载时检查是否有未保存数据
useEffect(() => {
  const unsaved = localStorage.getItem(`unsaved_${patternId}`)
  if (unsaved) {
    // 提示用户恢复
    offerRestore(JSON.parse(unsaved))
    localStorage.removeItem(`unsaved_${patternId}`)
  }
}, [])
```

### 六、性能细节

| 问题 | 解法 |
|------|------|
| 填充操作一次改几百格 | 整体作为一个 diff，一次写入 |
| 画笔连续拖动触发几十次 | `pointerdown` → `pointermove` 期间只更新内存，`pointerup` 才 push 到 history |
| 大尺寸图纸（50x50）存储慢 | pixels 只存非空格子，稀疏存储 |
| 读取慢 | 加载时一次性读入内存，后续全走内存 |

---

### 推荐组合

```
Zustand          内存状态 + 撤销栈
Dexie.js         IndexedDB 操作封装（比原生 API 好用很多）
防抖 800ms       合并高频写入
localStorage     关闭页面兜底
```

这套方案的核心原则是：**画布永远在内存里，IndexedDB 只是异步备份**，用户感知不到任何写入延迟。
