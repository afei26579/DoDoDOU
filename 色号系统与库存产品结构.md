# 色号系统与库存产品结构

## 目标

为拼豆图纸建立一套可扩展的色号系统，让用户可以在不同品牌色号之间切换、建立自己的常用色卡，并在图纸编辑、图纸生成、替代色选择和拼豆库存管理中复用同一套颜色数据。

最终希望解决三个问题：

1. 用户生成或编辑图纸时，可以按自己熟悉的品牌色号选色。
2. 用户看到图纸里没有的色号时，可以快速找到自己已有豆豆里的替代色。
3. 用户可以记录每个色号的库存，在拼图前判断是否够用，在拼完后按实际消耗扣库存。

## 现有数据基础

当前项目已经有两份核心色号数据：

- `色号对应表.csv`
  - 表头：`MARD, COCO, 漫漫, 盼盼, 咪小窝`
  - 当前约 291 行。
  - 每一行表示同一个颜色在不同品牌下的对应色号。
  - 适合作为“品牌间色号映射”的原始表。

- `colorSystemMapping.json`
  - 结构为 `hex -> 品牌 -> 色号`。
  - 每个 `hex` 对应一组品牌色号。
  - 适合前端生成调色板、图纸颜色匹配、导出色号统计。

当前代码中已经使用：

- `src/lib/pattern/color-system.ts`
  - `buildPalette(colorSystem)`
  - `getVendorCode(hex, colorSystem)`

- `src/lib/pattern/generator.ts`
  - 生成图纸时根据品牌色卡匹配颜色。

- `src/lib/pattern/download.ts`
  - 导出图纸时按指定品牌显示色号。

- `src/pages/workshop/WorkshopEditorPage.tsx`
  - 编辑器里根据颜色生成图纸结果和色号。

注意：需要检查色号数据的编码和字段名称。目前部分终端输出里中文品牌名出现乱码，正式实现前应统一为 UTF-8，并统一品牌 key。

建议标准品牌 key：

```ts
type BeadBrandKey = 'MARD' | 'COCO' | 'MANMAN' | 'PANPAN' | 'MIXIAOWO';
```

展示名称单独维护：

```ts
const brandLabels = {
  MARD: 'MARD',
  COCO: 'COCO',
  MANMAN: '漫漫',
  PANPAN: '盼盼',
  MIXIAOWO: '咪小窝',
};
```

## 产品结构

### 1. 品牌色卡

品牌色卡是系统内置数据，来自 `色号对应表.csv` 和 `colorSystemMapping.json`。

每个品牌色号应包含：

```ts
type BrandColor = {
  id: string;
  brandKey: BeadBrandKey;
  code: string;
  hex: string;
  rgb: {
    r: number;
    g: number;
    b: number;
  };
  name?: string;
  material?: 'normal' | 'transparent' | 'glow' | 'pearl' | 'special';
  enabled: boolean;
};
```

品牌色卡主要用于：

- 图纸生成时的颜色匹配。
- 编辑器里的品牌调色板。
- 导出图纸时显示品牌色号。
- 跨品牌色号查找。

### 2. 色号映射

色号映射表示不同品牌之间的同色或近似色关系。

当前 CSV 和 JSON 可以先认为同一行或同一个 `hex` 下的色号是“官方/整理映射”。

后续需要支持更细的替代关系：

```ts
type ColorAlias = {
  sourceBrandKey: BeadBrandKey;
  sourceCode: string;
  targetBrandKey: BeadBrandKey;
  targetCode: string;
  confidence: 'exact' | 'close' | 'usable';
  source: 'official' | 'system' | 'user';
  note?: string;
};
```

替代关系不应只做一对一，因为现实里会出现：

- 一个品牌色号可替代多个其他品牌色号。
- 两个颜色 `hex` 接近，但实物观感不完全一致。
- 用户自己认为某个颜色“可用”，但系统色差计算不一定认为最佳。

### 3. 我的色卡

我的色卡是用户自己的色号系统。

它不是一个全新的品牌，而是用户从多个品牌、自定义颜色、常用色里组合出来的工作色板。

```ts
type UserColorSet = {
  id: string;
  name: string;
  description?: string;
  defaultBrandKey?: BeadBrandKey;
  colorIds: string[];
  createdAt: string;
  updatedAt: string;
};
```

用户可以：

- 选择一个品牌作为默认色卡。
- 从多个品牌混合添加色号。
- 收藏常用色。
- 给颜色加别名，比如“肤色常用”“猫咪眼睛高光”。
- 隐藏自己不会用的色号。
- 在编辑器中优先展示自己的色卡。

### 4. 用户拥有的豆豆

库存系统建立在品牌色卡和我的色卡之上。

每条库存对应一个具体色号：

```ts
type BeadInventoryItem = {
  id: string;
  brandKey: BeadBrandKey;
  code: string;
  hex: string;
  quantity: number;
  lowStockThreshold?: number;
  location?: string;
  favorite?: boolean;
  note?: string;
  updatedAt: string;
};
```

库存系统用于：

- 判断图纸是否能直接开拼。
- 统计缺少哪些色号、每个缺多少颗。
- 推荐已有库存中的替代色。
- 拼完后扣除库存。
- 生成补货清单。

库存不是色号系统的必要前置条件。用户可以只使用色号系统，不录库存；但一旦录入库存，图纸统计和替代色推荐就可以变得更准确。

### 5. 图纸用色统计

每张图纸已经有 `PatternResult.palette`，后续需要增强为“用色需求”。

```ts
type PatternColorRequirement = {
  colorId: string;
  brandKey: BeadBrandKey;
  code: string;
  hex: string;
  requiredQuantity: number;
  ownedQuantity?: number;
  missingQuantity?: number;
  status: 'enough' | 'missing' | 'unknown';
};
```

生成图纸后，系统需要展示：

- 总颗数。
- 色号数量。
- 每个色号需要多少颗。
- 用户库存有多少颗。
- 是否足够。
- 缺多少颗。
- 可替代颜色。

### 6. 替代色方案

替代色方案用于把图纸里的某个色号替换成用户已有的另一个色号。

```ts
type PatternColorSubstitution = {
  source: {
    brandKey: BeadBrandKey;
    code: string;
    hex: string;
  };
  target: {
    brandKey: BeadBrandKey;
    code: string;
    hex: string;
  };
  reason: 'same-mapping' | 'nearest-color' | 'owned-stock' | 'user-defined';
  confidence: 'exact' | 'close' | 'usable';
  applied: boolean;
};
```

替代色推荐优先级建议：

1. 用户自定义替代关系。
2. 同一 `hex` 下的跨品牌映射。
3. 用户已有库存中 `hex/rgb` 最近的颜色。
4. 同品牌近似色。
5. 系统兜底推荐，但明确标记为“可能有色差”。

替代色应用后，图纸应重新计算：

- 图纸格子颜色。
- 色号统计。
- 库存缺口。
- 导出物料清单。

### 7. 库存扣减和消耗记录

扣库存应由用户确认，不建议生成图纸或进入专注模式时自动扣。

原因：

- 用户可能只是预览图纸。
- 用户可能只拼了一部分。
- 用户可能实际用了替代色。
- 用户可能拆掉重拼。
- 实际消耗可能比图纸多。

建议建立库存流水：

```ts
type InventoryTransaction = {
  id: string;
  inventoryItemId: string;
  type: 'add' | 'consume' | 'adjust' | 'restore';
  quantity: number;
  projectId?: string;
  patternId?: string;
  note?: string;
  createdAt: string;
};
```

支持操作：

- 手动增加库存。
- 手动调整库存。
- 按图纸一键扣除。
- 按已完成颜色扣除。
- 按专注模式进度扣除。
- 撤销一次扣除。
- 从图纸生成补货清单。

## 关键用户流程

### 流程一：用户选择默认品牌

1. 用户进入图纸参数设置。
2. 选择默认品牌，比如 `MARD`。
3. 系统使用 MARD 色卡生成图纸。
4. 图纸统计显示 MARD 色号。
5. 导出图纸默认显示 MARD 色号。

### 流程二：用户建立我的色卡

1. 用户进入“我的色卡”。
2. 选择常用品牌。
3. 添加自己常用的色号。
4. 可选：隐藏不拥有或不常用的色号。
5. 编辑图纸时优先展示我的色卡。

### 流程三：图纸生成后检查库存

1. 用户生成图纸。
2. 系统统计每个色号所需数量。
3. 系统读取用户库存。
4. 标记足够、缺少、未知。
5. 对缺少颜色给出替代色建议。

### 流程四：替换没有的色号

1. 用户在图纸统计中看到某色号缺少。
2. 点击“替换”。
3. 系统展示可替代色。
4. 用户选择一个自己拥有的色号。
5. 系统应用到整张图或当前选区。
6. 重新计算统计和库存缺口。

### 流程五：拼完扣库存

1. 用户进入专注模式拼图。
2. 完成部分或整张图纸。
3. 用户点击“扣库存”。
4. 系统展示将扣除的色号和数量。
5. 用户确认。
6. 生成库存流水，并更新库存。

## 发布计划

### P0：数据清理和基础模型

目标：先让系统拥有稳定、可维护的色号数据。

范围：

- 统一品牌 key，避免中文 key 在代码和 JSON 中乱码。
- 从 `色号对应表.csv` 和 `colorSystemMapping.json` 生成标准色卡数据。
- 保留当前 `hex -> brand -> code` 能力。
- 增加 `brandKey -> code -> hex` 查询能力。
- 给 `ColorSystem` 类型改成稳定英文 key，展示层再转中文名。
- 为缺失色号、重复色号、无效 hex 增加校验脚本。

验收：

- 所有品牌都能正常生成调色板。
- 图纸生成、编辑器、导出仍能显示正确色号。
- 不再出现因中文 key 编码导致的类型或运行时问题。

### P1：品牌色卡和编辑器选色增强

目标：用户可以在编辑图纸时更方便地按品牌色号选色。

范围：

- 编辑器调色板支持品牌切换。
- 支持色号搜索。
- 当前图纸用色单独展示。
- 图纸结果页显示完整品牌色号统计。
- 导出时可以选择显示哪个品牌的色号。

验收：

- 用户能从 MARD、COCO、漫漫、盼盼、咪小窝中切换色卡。
- 选择色号后，编辑器格子颜色和统计同步更新。
- 导出图纸和物料清单使用用户选择的品牌色号。

### P2：我的色卡

目标：用户可以配置自己的工作色号系统。

范围：

- 新增“我的色卡”管理入口。
- 支持从品牌色卡添加颜色。
- 支持常用色、隐藏色、别名。
- 编辑器调色板支持“我的色卡”模式。
- 图纸生成可选择优先使用我的色卡。

验收：

- 用户可以创建至少一套自己的色卡。
- 编辑器可以只显示我的色卡。
- 图纸生成时可以优先匹配我的色卡颜色。

### P3：库存管理

目标：用户可以记录自己每个色号有多少颗。

范围：

- 新增库存列表。
- 支持按品牌和色号录入数量。
- 支持库存搜索和筛选。
- 支持低库存阈值。
- 图纸结果页显示库存是否足够。
- 缺少色号显示缺口数量。

验收：

- 用户能录入和修改库存。
- 图纸生成后能判断每个色号够不够。
- 缺色清单能正确计算缺少数量。

### P4：替代色推荐和应用

目标：用户看到没有的色号时，可以使用已有豆豆替代。

范围：

- 图纸统计中增加“替换色号”入口。
- 推荐同 `hex` 跨品牌色号。
- 推荐用户库存中的近似色。
- 支持用户手动选择替代色。
- 替代后重新计算图纸统计和库存缺口。
- 支持保存用户自定义替代关系。

验收：

- 缺库存颜色可以看到可替代选项。
- 应用替代后，图纸和统计一致更新。
- 用户自定义替代关系下次继续生效。

### P5：扣库存、流水和补货清单

目标：拼完后库存能准确减少，并能追踪历史。

范围：

- 专注模式完成后支持扣库存。
- 图纸结果页支持按整张图扣库存。
- 支持扣除确认弹窗。
- 支持撤销最近一次扣除。
- 记录库存流水。
- 生成补货清单。

验收：

- 用户确认后才扣库存。
- 扣库存后库存数量和流水一致。
- 撤销扣除后库存能恢复。
- 补货清单能列出品牌、色号、缺少数量。

## 实现顺序建议

1. 先做数据清理和品牌 key 标准化。
2. 再把现有生成、编辑、导出统一接到标准色卡服务。
3. 然后做编辑器调色板增强。
4. 再做我的色卡。
5. 再做库存。
6. 最后做替代色推荐、扣库存和补货清单。

这个顺序可以避免一开始就把库存和替代色塞进生成算法里。色号系统稳定后，库存和替代色只是围绕图纸用色统计做增强，风险更低。

## 需要注意的产品边界

- 色号系统是基础能力，库存系统是增强能力。
- 没有录库存的用户，也应该能正常生成、编辑、导出图纸。
- 替代色不应静默自动应用，必须让用户确认。
- 扣库存不应自动发生，必须让用户确认。
- 图纸应该保留原始推荐色，同时记录用户替代后的实际用色。
- 用户自定义色号和替代关系优先级应高于系统推荐。

## 后续待确认问题

1. 是否允许用户新增完全自定义品牌？
2. 是否需要支持不同规格豆豆，比如 2.6mm、5mm？
3. 是否需要记录每包豆豆的购买信息和价格？
4. 是否需要按盒子、抽屉、位置管理库存？
5. 图纸替代色是默认应用到整张图，还是支持只替换选区？
6. 库存扣减是只按整张图，还是要和专注模式完成进度绑定？
7. 是否需要从 CSV 或表格批量导入用户库存？
